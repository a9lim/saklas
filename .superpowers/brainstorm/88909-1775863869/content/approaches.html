Three approaches for evolving steer

Pick the one that resonates, or mix and match

A

Science-First

Lead with new steering methods, let extensibility follow.

Add representation engineering (RepE), linear probing, per-layer sweep analysis, and vector arithmetic directly into the core. Build the plugin system around what these methods need — extraction protocols, hook interfaces, probe types. The API surface emerges from real use.

Pros

  • You get the science capabilities fastest
  • Plugin API is grounded in real requirements, not speculation
  • Each new method validates the abstraction

Cons

  • Plugin API may need rework as more methods reveal new patterns
  • Community contributors blocked until the API stabilizes
  • Risk of baking method-specific assumptions into core
B

Platform-First

Build the plugin/extension system first, then implement methods as plugins.

Define clean protocols: ExtractionMethod, ProbeType, MonitorHook, GenerationModifier. Refactor ActAdd and CAA into the first two plugins. New methods (RepE, linear probes, vector algebra) are plugins from day one. Headless API is just the plugin system without the TUI.

Pros

  • Contributors can add methods without touching core
  • Forces clean separation — no method-specific code in the engine
  • Headless/API mode falls out naturally

Cons

  • Risk of over-engineering abstractions before you know what methods need
  • Slower to get new science capabilities working
  • Plugin API designed in a vacuum may not fit real methods well
C

Interleaved (Recommended)

Define minimal protocols, implement 2-3 methods against them, then stabilize.

Start with lightweight protocols (Python Protocol classes — structural typing, no inheritance). Implement RepE and linear probing as the first two new methods against those protocols. Let the protocols evolve as the methods reveal what's actually needed. Stabilize the API after 3-4 methods prove the abstractions work. Headless mode and improved TUI monitoring land in parallel.

Pros

  • Protocols validated by real methods before they're locked down
  • New science capabilities land early
  • Contributors can start building once 2-3 methods prove the pattern
  • Avoids both premature abstraction and spaghetti-then-refactor

Cons

  • Protocols may change under early adopters (acceptable at v0.x)
  • Requires discipline to keep protocols minimal
