# tui/

Textual frontend over `SaklasSession`. Three panels — left (vectors), center (chat), right (monitor probes). 15 FPS poll loop (`_POLL_FPS = 15`).

The read-side probe paths run on the unified `Monitor` (the `a498895`
unification of the former `TraitMonitor` + `ManifoldMonitor`). Every attach is
`session.add_probe(selector, *, as_name=None, top_n=3)` and every detach is
`session.remove_probe(name)` over the one `session._monitor`; per-token readings
ride `event.probe_readings` and the end-of-gen aggregate is
`session.last_result.probe_readings` — both the single `ProbeReading` shape
(`coords`/`fraction`/`nearest`/`residual` + `*_per_layer`). The trait panel
splits the one probe set by geometry — flat (affine) probes drive the scalar
MONITOR PROBES section, curved probes the MANIFOLD PROBES section — the TUI
mirror of the webui's subspace/manifold rack split (`_refresh_probe_panels` is
the funnel; it keys on `core.manifold.manifold_is_affine`).

## Files

- `app.py` — `SaklasApp`; UI state, `BINDINGS`, `on_key`, generation workers, polling. The framework-dispatched surface (registry handlers, `action_*`, `on_*`) stays here; the steering/probe slash-command *logic* lives on `ExtractionController` and the loom slash/worker *logic* on `LoomController` (see below), with the App method names thin one-line forwarders into them.
- `extraction_controller.py` — `ExtractionController`; owns the TUI-local steering state (`_alphas` / `_enabled` / `_manifold_terms`) the session doesn't own, plus every handler that mutates it. App panel actions call narrow typed controller operations; the remaining App views are read-only mappings. `_active_alphas` / `_vector_list_for_panel` are pure derivations off that state and live here too.
- `loom_controller.py` — `LoomController`; owns the loom slash-command + worker *logic* (`_handle_tree` / `_handle_nav` / `_handle_edit` / `_handle_branch` / `_handle_del` / `_handle_star` / `_handle_note` / `_handle_path` / `_handle_fan` / `_handle_prune` / `_handle_auto_regen` / `_handle_diff`, plus the fan/regen worker bodies `_run_fan_worker` / `_run_regen_modifier_worker` / `_run_regen_n_worker` and the `_dispatch_loom_*` / `_render_node_diff` / `_handle_diff_siblings` / `_parse_custom_auto_regen` helpers) and the three stash attrs they read/write (`_loom_prune_expr` / `_loom_auto_regen_mode` / `_loom_auto_regen_on`). Composed onto the App via `self._app`. The App exposes the three stash attrs as read/write proxy properties so `_poll_generation` (the auto-regen routing + the status-footer dedupe tuple), `loom_screen.py`, and the test suite reach the same state. The `action_*` keybindings (`action_open_loom` / `action_nav_picker` / `action_edit_active` / `action_branch_active` / `action_delete_subtree`), the `commands.py` registry handlers, the `loom_screen.py` back-calls (`_dispatch_loom_regen` / `_dispatch_loom_fan`), and the `_dispatch_pending_action` drain all forward through thin App methods — so the worker bodies stay reachable on the App as `app._run_fan_worker` etc. for the tests that monkeypatch them. `_fire_auto_regen` is the one piece that does **not** move: it streams into the shadow column via the `_ui_token_queue`, so it's generation lifecycle and stays on the App with the rest of the gen orchestration, *reading* the controller's mode state through the proxies.
- `commands.py` — slash-command registry (`SlashCommand` table + `dispatch`). Slash handling lives here, not in `app.py`'s dispatch. Registry entries name `SaklasApp._handle_*` by attribute — those names are the forwarders above, so the registry is byte-unchanged by the controller split.
- `chat_panel.py` — `ChatPanel`, `ChatInput`, `_AssistantMessage`, `_TurnRow` (chat mode); `RawBuffer` + `_RawTextArea` (raw mode); per-token highlight markup. `ChatPanel._render_mode` (`"chat"`/`"raw"`) swaps the centre surface.
- `vector_panel.py` — `LeftPanel`; `MAX_ALPHA = 1.0`. Rows carry a `kind` discriminator: `vector` rows show an alpha bar, `manifold` rows show the fixed position (`% 0.3,0.8`) and blend coefficient instead.
- `pairs_modal.py` — `CustomPairsModal` (`ModalScreen`) for `/pairs`; `parse_pair_lines` splits the `positive | negative` buffer.
- `trait_panel.py` — `TraitPanel`; probe list + optional WORKSPACE (live J-lens readout, `/lens`) section + WHY/LAYERS histogram.
- `selectable.py` — `SelectableListWidget`; shared cursor (next/prev, `_cursor_wrap` wrap-vs-clamp, `_cursor_len` hook) + `selection_marker`/`name_markup` row markup for the two list panels (`LeftPanel`, `TraitPanel`).
- `loom_screen.py` / `loom_helpers.py` — full-screen loom navigator + formatting helpers.
- `utils.py` — `build_bar`, `BAR_WIDTH = 24` (every bar in the UI; the left panel's right-edge alignment `RIGHT_W = 11 + BAR_WIDTH + 4` is a function-local of `LeftPanel._render_gen_config`).

## app.py

Thin frontend. Owns thinking state and the gen-lifecycle / A-B / shadow / input-history / pending-queue orchestration; builds `SamplingConfig` + `Steering` from UI state per call. The local alpha/enabled/manifold steering state lives on `ExtractionController` (proxied back through the `_alphas` / `_enabled` / `_manifold_terms` properties); the loom slash/worker logic + the `_loom_prune_expr` / `_loom_auto_regen_mode` / `_loom_auto_regen_on` stash live on `LoomController` (proxied back through the matching `_loom_*` properties). `_dispatch_pending_action` stays on the App as the central `kind`→handler router (it fans out to the generation methods *and* the forwarders into both controllers — `fan` / `regen_n` route to the loom workers). `_fire_auto_regen` (the post-gen shadow-column sibling firing) stays on the App too, reading the loom controller's mode through the proxies each poll tick. Thinking defaults to whatever `supports_thinking(tokenizer)` reports; `Ctrl+T` toggles it.

The `_generate` worker is a thin wrapper over `session.generate_stream` — no direct reach into `_apply_steering` / `generate_steered` etc. It forwards events through an app-local, union-typed `_ui_token_queue: SimpleQueue[_UiQueueItem]`: `_UiToken`, `_UiFinalize`, `_UiError`, or `_UiDone`. `_poll_generation` drains the queue; the worker's `finally` emits `_UiDone` after session teardown.

Panel index constants: `_LEFT, _CHAT, _TRAIT = 0, 1, 2`.

## Slash commands

Registered in `commands.py::_build_registry`. `dispatch` validates the whitespace-token count against each entry's `min_args`/`max_args` and, for `interrupts=True` entries while `app._is_busy`, enqueues a `PendingItem` on `app._pending_queue` instead of running the handler immediately — the in-flight gen is left alone and the queued item drains on the next `done`.

- Steering: `/steer <expression>`, `/alpha <value> <name>`, `/unsteer <name|ns/>`.
- Probes: `/probe <selector|pos . neg|ns/>`, `/unprobe <name|ns/>`, `/extract <concept|pos . neg> [--role <slug>]` (cache-warm only). A single `/probe <selector>` attaches through the unified `session.add_probe` resolver, so flat concepts, curved manifolds, bundled names, and `ns/name` selectors share one front-end. The explicit `<pos> . <neg>` form first extracts the bipolar concept and then attaches its folded rank-1 probe. `/unprobe` detaches every probe shape via `session.remove_probe`.
- Custom extraction: `/pairs <name> [--role <slug>]` opens `CustomPairsModal` (`pairs_modal.py`) — a multi-line `TextArea`, one `positive | negative` pair per line, `Ctrl+S` (or `Ctrl+Enter`) extract / `Esc` cancel. Submitted pairs go straight to `session.extract_vector_from_corpora` (the corpus-in extractor — authors a 2-node `pca` manifold from the pos/neg lists), skipping scenario/pair generation. The modal can't be a `PendingItem`, so `/pairs` is refused mid-generation rather than queued.
- Role-augmented extraction: `--role <slug>` is peeled off `/extract` and `/pairs` args by `SaklasApp._peel_role_flag` before the bipolar parser runs (so multi-word poles don't collide with the trailing flag), and forwarded to `session.extract(role=...)`. A `:role-<slug>` variant typed directly on a `/steer` term (`/steer 0.5 angry:role-pirate`) also triggers a fresh role-augmented extraction when the tensor isn't cached — same shape as `:sae-<release>`. The variant suffix wins over an explicit `--role` flag because it rides through `canonicalize_atom`'s `explicit_variant` arm. Slug validation lives in `core.role_templates`; family unsupported (Mistral-3) raises `RoleSubstitutionUnsupportedError` at extract time.
- Manifold: `/manifold fit <folder>` fits an authored manifold pack via `session.fit` (gated like `/extract` — refused during A/B shadow, deferred behind an in-flight gen as a `manifold_fit` pending item). Manifold authoring stays out of the TUI. A `%` term routes through `_dispatch_manifold_term`, which validates it and registers a typed `ManifoldTerm` through `ExtractionController`; its `AlphaEntry` view is merged at generation time.
- Curved manifold probes use that same `/probe` command and pending `probe` action. Per-token readings (`event.probe_readings`, the full per-probe `ProbeReading` dict) ride `_UiToken.probe_readings`; `_poll_generation` pushes them to the trait panel via `update_manifold_readings(per_token=...)` (only curved probes render there), collapses each reading to coordinate axis 0 for the highlight markup, and `_finalize_widget_highlight` calls `_pull_manifold_aggregates` to read `session.last_result.probe_readings` and push the last-content-token aggregate. Shadow streams skip the per-token push so the steered rack isn't clobbered.
- Live lens: `/lens [off | L1,L2,...]` toggles the live all-fitted-layer J-lens readout (`session.enable_live_lens`/`disable_live_lens`; a missing lens artifact surfaces the `saklas lens fit` hint as a system message). The per-step `{layer: [(token, score)]}` payload uses the same K as the generation's logit alternatives and rides `event.lens_readout` → `_UiToken.lens_readout`; the layer-aggregated `[(token, strength, com, spread)]` chip list rides `event.lens_aggregate` → `_UiToken.lens_aggregate`. Both land in `TraitPanel.update_lens_readout(readout, aggregate=)`, rendering in the WORKSPACE section — a `Σ` row of `token@com` first (com = probability-mass-weighted depth center of mass, 0..1), then the per-layer rows (`set_lens_active(layers)` arms/hides it; shadow streams skip the push, same rule as the probe rack). `_handle_lens` lives on `SaklasApp` directly (session-level toggle, no controller state).

A bare slug typed into `/steer` (`/steer 0.7 pirate`) resolves in `core.steering_expr` through `io.selectors.resolve_bare_atom` — the single-owner tier ladder: manifold-label tier first (a node label, e.g. `pirate`), then the composite-name tier (a dotted 2-node `pca` manifold name), then pole canonicalization (`canonicalize_atom`). A unique manifold-label match synthesizes a label-form `ManifoldTerm` (`/steer 0.7 pirate` → `local/persona%pirate`). Cross-tier ambiguity raises `AmbiguousSelectorError`. Persona manifolds (per-node `role` set) also drive role-paired steering: at scope entry `session.steering()` aggregates each manifold term's `manifold.nearest_node_role(position)` and pipes the winning role through `session._active_role` so the prefill applies the chat-template substitution. Disagreement (multiple terms imply different roles) is soft-warn + highest-`|coeff|`-wins (`RoleBaselineMismatchWarning`); explicit `:role-<X>` on a vector term overrides any manifold-implied role with the same warning.
- Session: `/clear`, `/rewind`, `/regen [N] [mode]`, `/sys` (alias `/system`), `/temp`, `/top-p`, `/max`, `/seed [n|clear]`, `/save <name>`, `/load <name>`, `/export <path>`, `/model`, `/help`, `/exit` (alias `/quit`).
- Commit (no-generation send): `/commit <text>` — the cross-terminal fallback for the `Ctrl+Enter` / `Alt+Enter` binding. Routes through the same `_commit_with_text` helper, so the role-aware dispatch (user-node → authored assistant turn; otherwise → new user turn) and the mid-gen pending-queue handoff are identical.
- Analysis: `/compare <a> [b]` — 1-arg ranks cosine vs all loaded profiles; 2-arg is a pairwise score.
- Loom: `/tree`, `/nav <prefix>`, `/edit <text>`, `/branch [text]`, `/del [yes]`, `/star`, `/note <text>`, `/path`, `/fan <vector> <alphas>`, `/prune <filter-expr>`, `/auto-regen [mode]`, `/diff <id1> <id2> [--full]` / `/diff --siblings`.

`/steer` and friends take a full steering expression parsed by `saklas.core.steering_expr.parse_expr` — bare poles resolve through `resolve_bare_atom`'s manifold-label tier (a bipolar pole is a node label, lowered to a `%` push); variant suffixes (`:sae`, `:sae-<release>`, `:role-<slug>`), triggers (`@after`, `@when:…`), and manifold `%` terms are accepted. Projection (`a~b`, `a|b`) and ablation (`!x`) terms are **rejected** from `/steer` with an "express them in the YAML config" system message — they need session-level materialization the live `/steer` path doesn't run (`_handle_steer` rejects `ProjectedTerm`/`AblationTerm`). A bare term with no coefficient uses `DEFAULT_COEFF` (0.5). `/probe <name>` also seeds the highlight probe and turns highlighting on.

Namespace bulk forms (`/steer ns/`, `/probe ns/`, `/unsteer ns/`, `/unprobe ns/`): the trailing-slash form is caught by `_detect_namespace_selector` before the parser runs. Add-side enumerates `all_concepts()` (the manifold walk) for the namespace and routes through `session.ensure_profile_registered` (folds a fitted 2-node manifold — no fitting forward pass). `/steer ns/` registers loaded concepts at `DEFAULT_ALPHA = 0.5` with `enabled=False`. Concepts with no fitted tensor for this model get a one-line summary plus a re-extract hint.

## Keybindings

Chat-screen `BINDINGS` (`SaklasApp`): `Ctrl+Q` quit · `Backspace`/`Delete` remove vector · `Ctrl+A` A/B · `Esc` stop gen · `Ctrl+R` regen · `Ctrl+C` copy · `Ctrl+T` thinking · `Ctrl+S` sort · `Ctrl+Y` / `Ctrl+Shift+Y` highlight-mode cycle · `Ctrl+L` loom · `Ctrl+O` chat ⇄ raw render mode · `Ctrl+E` edit · `Ctrl+B` branch · `Ctrl+N` nav · `Ctrl+D` delete · `Ctrl+Enter` / `Alt+Enter` commit (no-gen send) · `[` `]` temp · `{` `}` top-p.

`Ctrl+O` toggles the render mode, not `Ctrl+M` — `Ctrl+M` transmits `0x0D` (CR), which Textual maps to `enter` before binding resolution, so a `ctrl+m` binding collides with the Enter key (the same byte collision documented below for `Ctrl+H` → backspace).

`Ctrl+Enter` and `Alt+Enter` are bound with `priority=True` so the chat input widget can't swallow them. `Alt+Enter` is the cross-terminal fallback — but on stock macOS Terminal.app and iTerm2 neither passes through without a config tweak: Terminal.app needs "Use Option as Meta key" enabled, iTerm2 needs the "Esc+" mode for the Option key. Modern terminals with the CSI-u / kitty keyboard protocol (Ghostty, Kitty, WezTerm) get both for free.

Most app-level `Ctrl+letter` shortcuts that collide with the `ChatInput` editor bindings (`Ctrl+A` → line-start, `Ctrl+C` → copy, `Ctrl+D` → delete-right, `Ctrl+E` → line-end, `Ctrl+Y` → redo) carry `priority=True` so the app shortcut wins when the input is focused. The non-colliding editor verbs TextArea ships with (`Ctrl+W` delete-word-left, `Ctrl+U` delete-to-line-start, `Ctrl+K` delete-to-line-end, `Ctrl+V`/`Ctrl+X`/`Ctrl+Z` clipboard + undo) keep working — nothing at the app layer claims them.

`on_key` handles the rest contextually. Input-focused: `Tab`/`Shift+Tab` switch panels; `↑`/`↓` recall input history (chat input only, *edge-only* — recall fires when the cursor sits on the first/last row of the multi-line buffer, otherwise the arrow falls through to TextArea's cursor nav). Panel-focused: `Tab` cycle panels, `↑`/`↓` nav within panel, `←`/`→` nudge alpha by `_ALPHA_STEP_FINE = 0.01`, `Shift+←`/`Shift+→` by `_ALPHA_STEP_COARSE = 0.1` (both clamp to `MAX_ALPHA`), `Enter` toggle vector. The shift-arrow variants live in `on_key`, not `BINDINGS`, because arrow handling is already context-gated.

Gotcha: `Ctrl+H` can never fire — terminals send `0x08`, which Textual hard-maps to `backspace` before binding resolution. Same byte collision hits `Ctrl+Shift+H` on terminals without shift reporting; this is why the highlight cycle is on `Ctrl+Y`.

## Highlight modes

Three-state cycle `{off → probe → surprise}` — `Ctrl+Y` forward, `Ctrl+Shift+Y` backward. There is no separate binary toggle and no `/surprise` command. The `probe` slot defers to whatever the trait panel has selected; with no probes loaded it's skipped and the cycle collapses to `{off ↔ surprise}`. Surprise mode is sentinelled by `chat_panel.SURPRISE_PROBE = "__surprise__"` on `_highlight_probe`. The active mode shows as the persistent `HL` line in the left panel's GENERATION block; `_apply_highlight_to_all` is the funnel that refreshes it.

Live probe scores stream only when probe highlighting is active (`_wants_live_probe_scores()`; surprise mode uses logprobs and returns false). Worker forwards `event.probe_readings` through `_ui_token_queue`; `_poll_generation` calls `widget.append_token_score(...)`. Surprise mode tints by `1 − exp(logprob) = 1 − probability` (`chat_panel._surprise_score`, unified with the webui `surpriseScore` in `webui/src/lib/tokens.ts`); the streaming `SamplingConfig`s set `logprobs=0` so `event.logprob` is populated. At finalize, canonical projected scores from `session.last_per_token_scores` overwrite live ones via `_finalize_widget_highlight`.

## Generation, prefill, interruption

Role-aware send: when the active loom node is a *user* turn, a typed message composes the assistant reply (answer-prefill) rather than a new user turn. `on_chat_panel_user_submitted` resolves `_prefill_target_node_id()` once at submit time and carries it in the `("submit", text, prefill_target)` pending tuple. Prefill routes to `_start_prefill` → `session.prefill_assistant`, streaming through the same `_ui_token_queue` (no live probe scores on this path). `chat_panel.set_prefill_mode(on)` flips the input placeholder; `_refresh_input_mode()` syncs it to the active node.

`chat_panel.on_input_submitted` does not mount the user row (it can't know the active-node role) — it just posts `UserSubmitted`; `on_chat_panel_user_submitted` mounts it for normal sends, skips it for prefills.

`Ctrl+Enter` / `Alt+Enter` is the commit (no-generation) path: `action_commit_text` reads the input value directly off the `#chat-input` widget, branches on `_prefill_target_node_id()` (user-role active node → `_start_commit_assistant` → `session.append_assistant_turn`; otherwise → `_start_commit_user` → `session.append_user_turn`), and mounts the row up front. Mid-gen commits enqueue a `PendingCommitUser(text)` or `PendingCommitAssistant(text, user_node_id)`; `_dispatch_pending_action` handles the closed pending-action union exhaustively. Commit workers don't stream — the row mounts finalized (via `chat_panel.add_finalized_assistant` for the assistant case, `add_user_message` for the user case) and the session method runs on a thread; on failure the optimistic row is removed and a system message is surfaced.

## Pending queue

While a generation is in flight every submission — chat send, commit, `Ctrl+R` regen, interrupting slash command (`/clear`, `/rewind`, `/quit`, `/regen N`, `/fan …`, `/steer …`, etc.) — appends one concrete member of the `PendingItem` discriminated dataclass union to `app._pending_queue` instead of stopping the gen. Each member carries named, typed fields rather than positional payload tuples. The pending strip (`saklas/tui/chat_panel.py::PendingStrip`) renders the live queue between the status bar and the chat input as a single dim line: `pending 3 · msg what do you think? · /steer /steer 0.5 angry · …`. The slot the user has pulled into the input via ↑ renders with a leading `✎` marker and undimmed yellow text so the editing target is visible without an extra row. `_poll_generation` drains one item per `_UiDone` event via `_drain_next_pending`; each drained item kicks its own gen whose own `done` advances the queue. `_sync_pull_state` is the single funnel for any `_pulled_slot` mutation — it flips `ChatInput.allow_empty_submit` and re-renders the strip in lockstep so the two derived surfaces never drift.

`Esc` is context-sensitive: a gen in flight → `session.stop()` (queue keeps its items, drain proceeds on the resulting `done`); no gen + a pending slot pulled → cancel the pull; otherwise no-op. No keypress interrupts the current gen *and* clears the queue — the user picks one.

Up-arrow walks the combined ring `[pending_queue (most-recent first), input_history (newest first)]`. Pulling a queued item sets `_pulled_slot` so the next `Enter` *replaces* that slot (slot-preserving edit), and `ChatInput.allow_empty_submit` flips on so an empty `Enter` (e.g. after `Ctrl+U`) *removes* the slot — the keyboard equivalent of the GUI's per-bubble `×`. `Esc` while pulled cancels the edit (slot stays as-is, input restores the stash). Non-interrupting slash commands (`/temp`, `/seed`, `/sys`, …) still run immediately even mid-gen — they don't conflict with the in-flight stream.

`_generate_core` calls `session._check_user_send_target(parent_node_id)` before `tree.add_user_turn` to refuse anchoring a user turn under a user-role node. Regen paths pass `parent_node_id=<user.parent_id>` explicitly so dedup reuses the existing user node.

## A/B and auto-regen

`Ctrl+A` (`action_ab_compare`) toggles both `_ab_mode` (the two-column layout) and `_loom_auto_regen_on` together — A/B is the default mode of the general auto-regen modifier. The chat log paints two columns per turn via the `_TurnRow` Horizontal, whose shadow column is `display: none` until the panel carries `.ab-on`.

`/auto-regen [mode]` stores `_loom_auto_regen_on` + `_loom_auto_regen_mode`. Built-in modes (`unsteered` / `inverted` / `reseed` / `cool` / `hot`) stash as strings; `custom: <expression>` parses into a `Recipe` partial. After each primary steered `done`, `_poll_generation` routes by mode: `unsteered` → `_start_shadow_generation` (stateless unsteered rebuild via `_build_shadow_messages`); any other mode → `_fire_auto_regen` (`generate_stream(..., recipe_override=mode)`). Both stream into the shadow column under `is_shadow=True`.

## Loom screen (`loom_screen.py`)

Full-screen `Screen`, reached via `Ctrl+L` (and `Ctrl+E`/`Ctrl+B`/`Ctrl+N`/`Ctrl+D`). Layout: one-line `#loom-header`, a `Tree` (left, `2fr`) / scrollable `#loom-detail` Static (right, `3fr`) split, and a hand-rendered two-line `#loom-keyhint` Static — *not* Textual's `Footer`, whose theme-variable rendering collided with the app's ANSI theme.

Navigation is cursor-based: `↑↓`/`kj` move through every visible node, `←→`/`hl` fold-then-parent / unfold-then-first-child, `Tab`/`Shift+Tab` jump siblings, `g`/`G` first/active node, `Enter` activates + returns to chat, `Space` activates and stays, `Esc` exits compare/help/filter then backs to chat. Mutations: `e` edit, `b` branch, `r` regen, `d` delete, `s` star, `a` note, `f` fan. Analysis: `c` flips the detail pane to turn-compare (`loom_helpers.format_compare` — diffs the assistant continuations of one user turn; the anchor is role-aware whether the cursor is on the user node or one of its replies), `/` search with `n`/`N`, `p` edit prune filter inline, `?` toggle keymap overlay.

Gotchas:
- `enter`/`space`/`tab`/`shift+tab`/`escape` are `priority=True` bindings — they'd otherwise be eaten by the focused `Tree` or Textual focus traversal. `check_action` disables all but `back` while a `_PromptOverlay` owns focus.
- `_PromptOverlay.on_mount` focuses its own input; querying for that input from `_open_overlay` races Textual's deferred mount.
- `_rebuild_tree` does a full rebuild on every mutation. Cursor re-seat is deferred via `call_after_refresh(self._finish_rebuild)` (inline re-seat races `Tree.clear`'s cursor reset). `_rebuilding` gates stray `NodeHighlighted` events during teardown; `_finish_rebuild` reads `_cursor_id` fresh so a keypress in the rebuild window isn't clobbered.

## Destructive-op confirm

`/del` requires `/del yes` to delete the active subtree; bare `/del` prints the hint. `Ctrl+D` routes through the same guard (`action_delete_subtree` → `_handle_del("")`) — no instant delete. The loom-screen `d` binding opens a `_PromptOverlay` requiring a typed `yes`. Both surfaces call `delete_subtree` directly: when the doomed subtree contains the active node the engine re-seats the active pointer on the surviving parent (root → fresh start) and emits the new `active_node_id` on the mutation event. `_handle_del` surfaces the new id in the post-delete system message; the chat-screen path still refuses up front when the active node is the root (nothing to delete).

## Persistence and chat repaint

`/save <name>` serializes the **entire** tree to `~/.saklas/conversations/<name>.json` via `LoomTree.save` (structure + text + recipes; per-token scores are not saved). `/load` reads it back, rewires the event bus + conflict hook, swaps into `session.tree`, resets monitor history, and repaints. There is no autosave. A model-id mismatch warns but does not refuse. (`saklas.core.transcript` YAML is unrelated — it backs the CLI `transcript run` verb only.)

`_repaint_chat_from_active_path` rebuilds the chat log to show only the loom tree's active path. Every navigation surface routes through it: `/nav`, `/load`, and loom-screen exit (`_return_to_chat`). Per-token probe scores aren't persisted on loom nodes, so navigated-to history renders without probe highlight; surprise highlight survives because logprobs ride along in node token rows. Loaded trees carry only `node.text` — plain finalized text.

`_AssistantMessage.set_static_content` populates a finalized message in one shot; it stashes the payload on `_pending_static` and applies it from `on_mount` because Textual mounts asynchronously. The highlight probe is seeded onto each widget before mount so navigated-to history comes up already tinted.

## Loom analysis commands

- `/fan <vector> <alphas>` — canonical alpha-grid sweep; routes through `session.generate_sweep`, reads sibling ids from the returned `RunSet.node_ids` so all siblings share one user-turn anchor.
- `/regen [N] [mode]` — N-way sibling regen; a trailing mode routes through `session.regen_with_modifier`.
- `/prune <filter-expr>` — parsed by `saklas.core.tree_filter.parse_filter`, stashed on `_loom_prune_expr`. `_rebuild_tree` dims non-matching nodes; both the loom footer and the chat status footer surface the active expression. Empty arg clears.
- `/diff <id1> <id2> [--full]` — ulid-prefix resolution (`resolve_node_prefix`), `session.diff_nodes`, word-level text diff + colored reading deltas. `/diff --siblings` diffs the active user-parent's assistant children.

## Panels

**Left panel** (`vector_panel.py::LeftPanel`): MODEL info, STEERING VECTORS, GENERATION block, KEYS reference. `_render_gen_config` is right-edge-aligned to `RIGHT_W = 11 + BAR_WIDTH + 4`: `Temp`/`Top-p` bars, then `Max`/`Think`/`Sys`/`HL` rows with trailing dim hint glyphs. The `HL` line is the persistent highlight-mode readout, set via `LeftPanel.update_highlight(mode)`, guarded against pre-mount calls.

**Monitor-probes panel** (`trait_panel.py::TraitPanel` — the class/file keep the `trait` name, but the panel header renders `MONITOR PROBES`). The one unified `session._monitor` feeds both sections; `app._refresh_probe_panels` splits the probe set by `core.manifold.manifold_is_affine` — flat (affine) probes to the scalar MONITOR PROBES section, curved probes to the MANIFOLD PROBES section below — the TUI mirror of the webui's subspace/manifold rack split. A probe shows in exactly one section.

The scalar MONITOR PROBES section renders two lines per flat probe — `> <name> <sparkline>` then `<bar> <val><arrow>`, the bar reading coordinate axis 0 (the pole-normalized coord for a 2-node concept). Full untruncated names; `_nav_items` is `list[str]`. The bottom WHY footer (`#why-header`, set to literal `LAYERS` by `update_why`) shows per-layer importance as a horizontal histogram bucketed into `HIST_BUCKETS = 16` groups (from `saklas.core.histogram`, shared with `cli manifold why`): the per-layer `||baked||` of the folded direction for a foldable rank-1 probe (a 2-node concept axis), falling back to the per-layer Mahalanobis share for a multi-axis flat fit (e.g. personas) that has no single baked direction. `_refresh_trait_why` reads the selected probe's `Manifold` off `session._monitor.manifolds`. Driven on trait nav / probe add-remove / finalize / clear / mount — not per streamed token. No `/why` command.

The optional MANIFOLD PROBES section (hidden when no curved probe is attached) renders each curved probe as a name header, a `subspace_fraction` bar (`build_bar` at `BAR_WIDTH`) reading the aggregate's `fraction` (or the latest live `fraction` mid-gen; per-token and aggregate are the one `ProbeReading` shape now), the top-3 nearest node labels with their distances, and — for 2-D `BoxDomain` manifolds only (an authored 2-D box; no *bundled* manifold currently qualifies — `emotions` is 3-D, `personas` 8-D) — a `MINIMAP_W x MINIMAP_H` (`17 x 7`) ASCII mini-map that plots node positions as `·` and the inferred `coords` dot as `●` in `ansi_cyan`. The mini-map skips higher-dim manifolds (`emotions`'s 3-D and `personas`'s 8-D layouts have no meaningful rectangular projection) and falls back to label-only for `SphereDomain` / `CustomDomain`. `set_active_manifold_probes(name → Manifold)` registers; `update_manifold_readings(per_token=, aggregates=)` pushes data (both `ProbeReading` dicts straight off the monitor — extra flat entries are ignored); `clear_manifold_readings()` resets on `/clear`. Aggregate beats live for the bar / nearest list — it's the reading pooled at the last-content token (bit-identical to the live read at that token), the canonical end-of-gen value.

**Render modes — chat vs raw**: `SaklasApp.__init__` reads `session.is_base_model` into `_is_base_model` and seeds `_render_mode` (`"raw"` for a base model, `"chat"` otherwise); base models also force-suppress thinking (`_supports_thinking` held False). `on_mount` calls `chat_panel.set_render_mode(_render_mode)`. `Ctrl+O` (`action_toggle_render_mode` → `_set_render_mode`) toggles at runtime — neither mode is locked to the model, so an instruct model can be driven as a raw completion model and vice versa. The toggle stops any in-flight gen, drops A/B (a chat-only layout), and repaints the now-visible surface from the loom tree.

Raw mode is `chat_panel.RawBuffer` — the TUI mirror of `webui`'s `RawBuffer.svelte`. The loom active path is flattened (`tree.flat_text()`) into one editable `_RawTextArea`; `Enter` posts `RawBuffer.Continue`, `Shift+Enter`/`Ctrl+J` insert a newline. `RawBuffer` implements the `_AssistantMessage` *streaming protocol* (`append_token`, `append_token_score`, `set_token_data`, `apply_highlight`, …) so `_poll_generation` routes a generation into it as the stream `widget` unchanged. `_start_generation(..., raw_continuation=True)` is the raw path: the divergence between the edited draft and the committed tree is resolved by `_resolve_raw_divergence` (port of the webui's `resolveDivergence`) into a `(tail, parent_node_id)` pair fed to `generate_stream(input=tail, raw=True, parent_node_id=parent)`. `Ctrl+Enter` commits the current divergent span without decoding (`append_user_turn(..., allow_any_parent=True)`), and `/commit <text>` appends + commits that text from the command line. Per-token highlight survives via a tinted `Static` mirror layered (CSS `layers: base over`) over the textarea, shown only while the buffer is clean — exactly the webui `showTint` rule. The chat input stays visible in raw mode as the command line: a `/`-prefixed line runs a slash command; a plain line appends + continues. `raw=self._raw_mode` rides on every `generate_stream` call, and raw regen passes `raw=True`, so the engine feeds the model `tree.flat_text()` verbatim rather than the `build_chat_input` `User:/Assistant:` marker fallback. Dirty mid-buffer edits now block navigation / mode-switch until committed or continued; inline tree mutations (`/edit`, `/branch`, `/del`) re-sync the raw buffer immediately. Remaining rough edge: A/B's two-column shadow layout is chat-mode only, though named/custom auto-regen can still fire row-less raw siblings. The loom screen still shows engine role tags — out of scope.

**Chat panel** (`chat_panel.py`): Markdown-free — user + assistant messages are plain `Static` widgets with Rich-escaped text. `_AssistantMessage` holds two content `Static`s (`#thinking-view`, `#response-view`) plus a `Collapsible` for thinking. Tokens escaped once on append. `append_token_score` adds live scores; `set_token_data` overwrites with canonical projected scores at finalize. Highlight saturation is fixed at `_HIGHLIGHT_SAT = 0.5`; zero scores get no background span. `_render_response` lstrips leading whitespace and `_build_highlight_markup` skips leading whitespace-only tokens (models emit `\n\n` after `</think>`). Thinking-block CSS zeroes Textual's `Collapsible` padding; transitions go through idempotent `ensure_thinking_collapsed()`.

**Chat input** (`ChatInput` in `chat_panel.py`): multi-line `TextArea` subclass. `Enter` submits via `ChatInput.Submitted` (re-posted by `ChatPanel.on_chat_input_submitted` as `ChatPanel.UserSubmitted`); `Ctrl+J` *and* `Shift+Enter` both insert a literal newline. Two newline keys because `Shift+Enter` is the natural chat-UI convention but only reaches the app on terminals with the CSI-u / kitty keyboard protocol (Ghostty, Kitty, WezTerm); stock macOS Terminal.app and iTerm2 silently collapse it to bare `Enter`. `Ctrl+J` sends a literal LF (0x0A) which Textual's `KEY_ALIASES` routes as a distinct key from `Enter` (CR, 0x0D) on every terminal, so it's the reliable cross-terminal fallback. All three live in an `_on_key` override — TextArea's default `_on_key` only treats bare `enter` as a newline insert, so `shift+enter` / `ctrl+j` would otherwise be dropped (or, in the case of bare `enter`, would insert a newline instead of submitting). `highlight_cursor_line=False` is passed to suppress the per-line "$boost" background TextArea otherwise paints under the cursor — under the ansi theme it read as an opaque black band; CSS also pins `.text-area--cursor-line` to `ansi_default` as defence in depth. `placeholder` carries the newline-key cheat sheet. CSS sizes the widget `height: auto` with `min-height: 3`, `max-height: 10` so the buffer grows with content up to a soft cap.

## Status footer

`chat_panel.update_status`: one-line footer — dot + `gen_tokens/max_new_tokens` + progress bar (green while generating, dim between runs, `○ idle` before the first gen) · tok/s · elapsed · `ppl <mean>`. `prune_expr` and `auto_regen_mode` kwargs append `· filter:<expr>` / `· auto:<mode>`. Perplexity is the geometric mean of per-step `TokenEvent.perplexity` (`_log_ppl_sum` / `_ppl_count`, reset at `_start_generation`).

## Input history

Shell-style recall on the chat input. Every submitted line lands in `_input_history` via `_push_input_history` (readline-flavored — `A→B→A` records both, `A→A→A` collapses). Capped at `_INPUT_HISTORY_MAX = 200`. `↑`/`↓` call `_history_navigate(∓1)`; the first `↑` stashes the in-progress draft, `↓` past the newest restores it. In-memory, process-scoped. Implemented in `on_key`'s input-focused branch, gated on `focused.id == "chat-input"` so panel-side `↑`/`↓` (trait nav) keeps its meaning elsewhere.
